# Context Nugget — Comprehensive Audit

**Date:** 2026-07-10
**Scope:** Full source review of `src/` (18 modules), `tests/`, `examples/`, `recipes/`, `design.md`, `README.md`, and packaging (`package.json`, `tsconfig.json`).
**Method:** Line-by-line code review, local build + test run (`npm test`: 8/8 pass on Node 20+), and empirical verification of every suspected defect with runnable repro scripts. This audit incorporates and verifies the earlier external review (`Context Nugget Review.txt`, 2026-07-09), which was a source-only audit without a local test run.

---

## 1. Executive summary

Context Nugget is a well-conceived seed. The core abstraction — *structured, inspectable context packets before prompt strings* — is genuinely useful, the boundary discipline (no model calls, no prompts, no storage policy) is consistently held, and the code is clean, dependency-free, strict-mode TypeScript that builds and passes its tests.

It is **not yet correct enough for its own headline claims**. The most serious problems cluster in one theme: **the library has no delete/replace lifecycle, so stale data outlives its source of truth**. Updated sources leave old chunks searchable; updated memories leave both versions retrievable; expired and superseded memories keep surfacing through their chunks even after `listMemories` correctly hides them. For a library whose pitch is *auditable, trustworthy context*, serving provably stale or expired context is the defect class to eliminate first.

Secondary themes: policy hooks that are declared but never enforced, citations whose line ranges silently disappear, diagnostics that misreport search scope, and a trust boundary that untrusted content can forge its way out of. All are fixable with modest effort; none undermine the architecture.

**Verdict (consistent with the prior review):** promising seed, coherent API, wrong to publish as-is. Fix lifecycle + policy enforcement + diagnostics, add CI and a truthful docs pass, then it is a credible v0.2.

## 2. Verification of the prior audit's findings

Every finding from `Context Nugget Review.txt` was checked against the code and, where possible, reproduced empirically.

| P2 | `MemoryPolicy.shouldRetrieve` / `shouldExpire` defined but never enforced | **Confirmed** | `types.ts:88-89` declares the hooks; `engine.ts:72-98` retrieves via `store.listChunks` and never consults `memoryPolicy`. Only `shouldStore` is ever called (`memory.ts:57-62`). |

Note the hardcoded `markdownChunker()` also discards any `TextChunkerOptions` defaults the app configured. |
| P4 | Diagnostics understate search scope (`searchedChunks` = result count, not candidates) | **Confirmed** | `pack.ts:61` sets `searchedChunks: results.length`, where `results` is the already-retrieved, ranked, top-K set. | P5 | Safety helpers exist but redaction is not wired into ingestion/retrieval/packing | **Confirmed** | `redactText` (`safety.ts:11`) has zero call sites outside its own module. `wrapUntrustedSourceData` is wired (`pack.ts:104`); redaction is not, and the README does not state this. |

| P7 | Store too naive for production memory use (no delete/replace/reindex/compaction) | **Confirmed** | `ContextStore` (`types.ts:186-196`) has only add/list/export/import/clear. No `removeSource`, `removeChunks`, `removeMemory`, or update semantics. |
| P8 | Docs slightly ahead of implementation (policy hooks, lifecycle) | **Confirmed** | README line 26 claims "approval-policy hooks" (only the store-side hook works); design.md implies reversible memory lifecycle that the store cannot express. |

All nine prior findings hold. Its "What Works Well" assessment (clean boundary, coherent API, zero runtime deps, targeted tests, good positioning wedge) also holds and is not repeated in detail here.

## 3. New findings (this audit)

Findings are ordered by severity. IDs `N1…N12` are referenced by the implementation plan. Items marked **(reproduced)** were demonstrated with a runnable script against the built `dist/`.

### High severity

**N1 — Expired, archived, and superseded memories remain retrievable through their chunks.** *(reproduced)*
`memoryStore.ts` applies `recordIsActive` (status + `expiresAt`) only in `listMemories` (`memoryStore.ts:17-23`). Retrieval, however, flows through `listChunks` (`engine.ts:73`), which knows nothing about memory status. Repro: a memory with `expiresAt` in 2020 returns `listMemories → 0` but `listChunks → 1`, so the engine happily packs expired memory into model context. This compounds P1/P2: even the lifecycle controls that *do* exist (`status: 'archived' | 'superseded'`, `expiresAt`) are dead on the retrieval path. For a memory library this is the single most misleading behavior — an app that "deletes" a memory by archiving it will keep leaking it to the model.

**N2 — The untrusted-source-data boundary is forgeable by the content it wraps.** *(reproduced)*
`wrapUntrustedSourceData` (`safety.ts:18-28`) uses fixed sentinels (`== BEGIN/END UNTRUSTED SOURCE DATA ==`) and does not escape occurrences of those sentinels inside the wrapped text. A retrieved document containing the literal `== END UNTRUSTED SOURCE DATA ==` followed by injected instructions produces packed output in which the injection sits *outside* the fence, indistinguishable from trusted packer output. Since this feature exists specifically for adversarial content (see `recipes/untrusted-repo-context.md`), sentinel forgery defeats its purpose. Fix: neutralize sentinel-like lines in wrapped content and/or use a per-pack random nonce in the delimiters.

**N3 — Line ranges are silently lost for realistic text, and wrong when present.** *(reproduced)*
`lineRangeForText` (`chunk.ts:38-45`) locates a chunk by `indexOf` of its first 80 characters in the original content. Two failure modes:
- `textChunker` builds chunk text by joining words with single spaces (`chunk.ts:31`), so for any source containing newlines or multiple spaces the search string never matches: **every chunk gets `lineStart/lineEnd: undefined`**. Repro: a 30-line document chunked with `textChunker` yields `[undefined, undefined]` for all 8 chunks.
- With `overlapWords > 0`, `nextSearchStart` advances past the *entire* previous chunk (`chunk.ts:44`), but the next chunk begins *inside* the previous one, so even matchable chunks fail from chunk 2 onward.
- When a match is found, `lineEnd = lineStart + chunkText.split('\n').length - 1` (`chunk.ts:43`) counts newlines in the *reconstructed* chunk text (usually zero), so a chunk spanning 15 source lines reports a single-line range.

Net effect: the "cited, traceable" line ranges the README and citation labels (`citations.ts:12-14`) promise are absent or wrong except in the narrow markdown-section happy path exercised by the test suite. Citations still render (falling back to title), so nothing fails loudly.

**N4 — `strategy` overrides silently discard the app's configured retriever and its options.**
`engine.retrieve` (`engine.ts:74-77`) replaces the configured retriever with a *fresh default instance* whenever `strategy` is `'keyword' | 'hybrid' | 'bm25'`. An app that configured `bm25Retriever({ k1: 1.2, b: 0.9 })` and then passes `strategy: 'bm25'` (matching what it configured!) gets default parameters instead. Combined with P3, the two main constructor injection points (`chunker`, `retriever`) both have paths that ignore them — the dependency-injection story is currently decorative.

### Medium severity

**N5 — Hybrid fusion adds raw BM25 and keyword scores on incompatible scales.**
`hybrid.ts:23` computes `existing.score += result.score`. BM25 scores are unbounded (typically 1–15 here); keyword scores are normalized to roughly 0–1.5 (`keyword.ts:19`). The keyword signal is therefore near-irrelevant except as a tiebreaker, and the "hybrid" mode is effectively BM25 with noise. Use reciprocal-rank fusion (RRF) or min–max normalization per retriever before combining.

**N6 — `RankOptions.maxItemsPerSource` is accepted and ignored.**
`engine.ts:88` passes `{ maxItemsPerSource: query.budget?.maxItemsPerSource }` into `rankResults`, and `RankOptions` (`types.ts:203-206`) declares the field, but neither `applySourceDiversity` nor `applyMemorySignals` reads it (`rank.ts`). The behavior users see comes solely from the budget stage (`budget.ts:15,29`). Dead options on a public type mislead integrators; either implement per-source capping at rank time or remove the field.

**N7 — 32-bit content hashes for chunk IDs will silently drop chunks at scale.**
`makeId` uses FNV-1a 32-bit (`util.ts:1-13`) over `sourceId:index:first-N-chars`. The store keys chunks by ID in a `Map` (`memoryStore.ts:35`), so a hash collision *silently replaces* an unrelated chunk. Birthday bound: ~50% collision probability around 77k chunks — reachable for a workspace-indexing use case. Fold in more entropy (full text length + full-text hash, or a 64-bit/128-bit hash such as FNV-1a-64 or cyrb128) while keeping determinism.

**N8 — Publishing as configured would ship a broken or stale package.**
`files` includes `dist/**/*.js` but there is no `prepublishOnly`/`prepare` build hook, and `dist/` is untracked — `npm publish` from a fresh clone ships no code, and publish after local edits ships whatever was last built. `declarationMap: true` emits `.d.ts.map` files that reference `../src` paths excluded from the package (broken go-to-definition for consumers). `package.json` lacks `repository`, `bugs`, and `homepage`. `examples/` are included in `files` but import `@jxburros/context-nugget` which is not installable yet, so they fail for anyone who tries them.

### Low severity / polish

**N9 — Secret redaction has significant pattern gaps.** *(reproduced)*
`redactText` misses AWS access keys (`AKIA…`), Slack tokens (`xox[baprs]-…`), PEM private-key blocks, GitLab PATs, and generic JWTs. Repro: a string with an AWS key, Slack token, and `-----BEGIN RSA PRIVATE KEY-----` passes through unchanged. Acceptable only if the docs say "best-effort, bring your own redactor" — they currently say nothing (see P5).

**N10 — `suggestMemory` policy dispatch has surprising precedence and dead code.**
`shouldStoreMemory` (`memory.ts:57-62`) returns early for `mode: 'manual'`, making `manualMemoryPolicy.shouldStore` (`memory.ts:4-7`) unreachable. For `mode: 'suggested'`, a user-supplied `shouldStore` is consulted *before* the "suggested requires app approval" branch, so a `suggested` policy with a `shouldStore` behaves identically to `auto`. The mode/hook interaction needs a documented, tested contract.

**N11 — Packet items leak full chunk metadata.**
`packetFromResults` (`pack.ts:37-41`) spreads all chunk metadata (which itself spreads all *source* metadata, `chunk.ts:65,165`) into `ContextItem.metadata`, plus memory internals (`scope`, `tags`, `importance`). None of it reaches the packed *text*, but packets are designed to be logged/inspected/exported, so apps that attach private fields to source metadata (the README's own example attaches `project` and `path`) get them replicated into every packet item. Consider an allowlist or a `metadataPolicy` option.

**N12 — Assorted small correctness notes.**
- `memoryStore.addSource` stamps `updatedAt` when absent, but `import()` does not apply the same normalization — snapshot round-trips are not identity-stable (`memoryStore.ts:31` vs `66-71`).
- Markdown section `startLine` is computed before the section text is `trim()`ed for matching (`chunk.ts:150-157`), so pieces after leading blank lines can be off by a line — worth covering once N3's rewrite lands.
- `chunkMatchesQuery` scope filtering relies on `chunk.metadata.scope` (`memoryStore.ts:13`), which only memory chunks set; a `scope`-filtered query silently excludes *all* document chunks. Reasonable, but undocumented.
- Empty/stopword-only queries return empty packets (fine) with `retrievalMode` unchanged and no diagnostic reason — add an explicit reason string and a test.

## 4. What works well (endorsed)

- **The packet-before-prompt abstraction and the boundary discipline** — retrieval/citation/budget/packing without model calls — remain the product's strongest ideas. Nothing in this audit suggests changing the architecture.
- **Zero runtime dependencies**, strict TS, `noUncheckedIndexedAccess`, ESM with proper `exports` map: the engineering baseline is genuinely good.
- **BM25 implementation is correct** (verified: standard Okapi weighting with the +1 idf floor, deterministic tie-breaking by ID) and the term-contribution "reasons" are a nice explainability touch.
- **Budget enforcement is correct and well-shaped** (`budget.ts`): greedy in rank order, per-source caps, reserve tokens, included/excluded reporting.
- **Degraded-mode semantics** (visible fallback instead of silent failure or throw) is the right call and is actually implemented for the semantic case.
- **The recipes are an asset**: they document intended usage patterns and boundaries better than most seed projects' docs.

## 5. Suggestions for improvement

Grouped by theme; the implementation plan (`docs/implementation-plan.md`) sequences these into phases with acceptance criteria.

### A. Data lifecycle (fixes P1, P7, N1)
1. Extend `ContextStore` with `removeSource(id)`, `removeChunks(filter: { sourceId?; memoryId? })`, `removeMemory(id)`, and `updateMemory(record)`.
2. Make `engine.addSource` **replace** semantics: remove existing chunks for `source.id` before adding new ones. Same for `addMemory` via a `memoryId` chunk filter.
3. Filter memory-backed chunks by record status/expiry inside `listChunks` (the store already owns the records), so archived/superseded/expired memories can never surface.
4. Add `engine.removeSource` / `engine.removeMemory` conveniences that keep chunks in sync.

### B. Policy enforcement (fixes P2, N10)
5. Apply `memoryPolicy.shouldExpire` and `shouldRetrieve` in `engine.retrieve` for memory-backed results, with drops recorded in diagnostics reasons.
6. Define and test the `mode` × `shouldStore` precedence contract; delete or repurpose the dead `manualMemoryPolicy.shouldStore`.

### C. Chunking & citations (fixes P3, N3, N12)
7. Respect the configured chunker for all source kinds; select markdown behavior via a `chunkerByKind` map (defaulting `markdown → markdownChunker(defaults)`) so injection always wins.
8. Rewrite line-range computation to track character offsets during chunking (precompute newline positions; map word/paragraph offsets to lines) instead of re-searching with `indexOf`. Line ranges must survive overlap, repeated text, and whitespace normalization — and `lineEnd` must reflect the *source* span.

### D. Diagnostics & packet honesty (fixes P4)
9. Report `candidateChunks`, `retrievedResults`, `returnedItems`, `excludedItems` distinctly; keep `searchedChunks` as a deprecated alias for one release.

### E. Safety hardening (fixes P5, N2, N9)
10. Neutralize sentinel-like lines inside wrapped untrusted text and add an optional per-pack nonce to the fence.
11. Expand `SECRET_PATTERNS` (AWS, Slack, PEM blocks, GitLab, npm, JWT-shaped) and add an opt-in `PackOptions.redact?: boolean | ((text: string) => string)` applied per item at pack time.
12. Write `docs/security-model.md`: what the library guarantees (structural boundary, best-effort redaction *when enabled*), what it never guarantees (prompt-injection immunity, PII discovery), and what stays app-owned.

### F. Retrieval quality (fixes P9, N4, N5)
13. `strategy` should select among *configured* retrievers (a `retrievers?: Record<string, Retriever>` option) rather than instantiating defaults; fall back to defaults only when nothing is configured.
14. Replace hybrid score addition with RRF (rank-based, scale-free) and keep per-retriever breakdowns.
15. Publish a `SemanticRetriever`/`Embedder` adapter contract (types only, no implementation) so the degraded-mode machinery has a real target.
16. Optional: cache the BM25 index keyed on a store revision counter; make `getChunk` a Map lookup. Low urgency at seed scale, cheap to do.

### G. IDs & scale (fixes N7)
17. Widen `stableHash` to 64+ bits (still dependency-free) and include full-text length in the seed; document determinism guarantees.

### H. Packaging, CI, release (fixes P6, N8)
18. GitHub Actions: install → typecheck → build → test → `npm pack --dry-run` on Node 20/22.
19. `prepublishOnly: npm run typecheck && npm run build && npm test`; add `repository`/`bugs`/`homepage`; fix or drop `declarationMap`; make examples runnable (per-example `package.json` with a `file:` dependency, exercised in CI).
20. Release checklist + provenance (npm `--provenance` once public).

### I. Documentation truth pass (fixes P8)
21. After B lands, re-verify every README/design claim ("approval-policy hooks", "reversible memory lifecycle", redaction wording per E12). Adopt the prior review's positioning line — *"auditable, cited, budgeted context packets for AI apps"* — as the README tagline.

### J. Test expansion (supports everything above)
22. Add tests for: source update/replace, memory update/expiry/supersession at retrieval time, custom-chunker respect, strategy-vs-configured-retriever, line ranges with overlap/newlines/repeats, sentinel forgery, redaction patterns and opt-in, diagnostics counts, empty-query behavior, scope/layer filtering on non-memory chunks, snapshot round-trip identity, and hybrid fusion ordering.

### Marketability (carried from the prior review, endorsed)
The narrow pitch — a lightweight TypeScript SDK for **auditable, cited, budgeted context packets** — is the right wedge; avoid "RAG framework" or "AI memory" framing that invites comparison with LangChain/LlamaIndex/Mem0/Zep. This audit adds one supporting argument: the audit-focused features (diagnostics, citations, trust boundary) are precisely the ones with the defects found above, so fixing sections C/D/E is not just correctness work — it is *the product*.
